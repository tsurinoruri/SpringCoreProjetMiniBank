    Модели данных
User (Пользователь)

id (int) - уникальный идентификатор

login (String) - логин пользователя

accountList (List<Account>) - список счетов

Account (Банковский счёт)

id (int) - уникальный идентификатор счёта

userId (int) - ID владельца

moneyAmount (int) - текущий баланс

Доступные команды
USER_CREATE - создание нового пользователя
Запрашивает логин, проверяет что пользователь не существует, создаёт пользователя с автоматическим ID и одним счётом с начальным балансом 500.

SHOW_ALL_USERS - отображение всех пользователей
Выводит список всех пользователей с информацией об их счетах.

ACCOUNT_CREATE - создание нового счёта
Запрашивает ID пользователя, создаёт новый счёт с балансом 500.

ACCOUNT_DEPOSIT - пополнение счёта
Запрашивает ID счёта и сумму, пополняет счёт.

ACCOUNT_WITHDRAW - снятие средств
Запрашивает ID счёта и сумму, проверяет достаточность средств.

ACCOUNT_TRANSFER - перевод средств
Запрашивает ID счёта отправителя, ID счёта получателя и сумму. При переводе между разными пользователями взимается комиссия 2%.

ACCOUNT_CLOSE - закрытие счёта
Запрашивает ID счёта, переводит остаток на первый оставшийся счёт пользователя. Нельзя закрыть единственный счёт.

EXIT - завершение работы программы

Настройки приложения
Файл application.properties содержит:

account.default-amount=500 - начальный баланс при создании счёта

account.transfer-commission=0.02 - комиссия за перевод между разными пользователями (2%)
При запуске программа показывает приветствие и список доступных команд.

Пользователь вводит команду user_create, затем логин Alice. Система создаёт пользователя с ID=1 и автоматически создаёт для него счёт №1 с балансом 500.

Пользователь создаёт второго пользователя Bob с ID=2 и счётом №2 с балансом 500.

Команда show_all_users показывает всех пользователей и их счета.

Пользователь вносит 300 на счёт №1 командой account_deposit. Баланс становится 800.

Пользователь переводит 200 со счёта №1 на счёт №2 командой account_transfer. Со счёта №1 списывается 200, на счёт №2 зачисляется 196 (с учётом комиссии 4). Баланс счёта №1 становится 600, счёта №2 - 696.

Команда exit завершает работу программы.

Как это работает
Ввод пользователя:
ConsoleListener отвечает за чтение команд с консоли. Он валидирует ввод, проверяет что команда не пустая и существует в списке доступных команд. Все методы readPositiveInt и readRequiredString защищают от некорректного ввода.

Регистрация команд:
Spring автоматически находит все классы, реализующие интерфейс OperationCommand и имеющие аннотацию Component. OperationsConsoleListener получает список всех таких команд через конструктор и преобразует их в Map для быстрого доступа.

Главный цикл:
OperationsConsoleListener запускает бесконечный цикл, который читает команду от пользователя, находит соответствующий обработчик в Map и выполняет его. При вводе команды EXIT цикл прерывается и программа завершается.

Бизнес-логика:
UserService и AccountService содержат всю бизнес-логику. Они хранят данные в коллекциях Map и генерируют уникальные ID для новых сущностей. При создании пользователя автоматически создаётся его первый счёт. При переводе проверяется принадлежность счетов разным пользователям для расчёта комиссии.

Обработка ошибок
Программа корректно обрабатывает все ошибки и выводит понятные сообщения:

Попытка создать пользователя с существующим логином - ошибка "User already exist"

Обращение к несуществующему пользователю - ошибка "No such user with id"

Обращение к несуществующему счёту - ошибка "No such account"

Ввод отрицательной суммы - ошибка "amount must be > 0"

Попытка снять больше чем есть - ошибка "insufficient funds"

Попытка закрыть единственный счёт - ошибка "Can't close the only one account"

Все ошибки не прерывают работу программы, пользователь может продолжить ввод команд.

Поток данных
Пользователь вводит команду в консоль. ConsoleListener валидирует ввод и преобразует строку в enum ConsoleOperationType. OperationsConsoleListener находит соответствующую команду в Map и вызывает её метод execute. Команда запрашивает у ConsoleListener дополнительные данные, затем вызывает методы UserService или AccountService. Сервисы выполняют бизнес-логику и возвращают результат. Команда выводит результат через System.out. Цикл повторяется.

Особенности реализации
Почему модели User и Account не Spring бины: Spring бины существуют в единственном экземпляре и управляются контейнером. Пользователей и счетов может быть много, они создаются динамически, поэтому они создаются через оператор new.

Почему команды отдельные классы: Это позволяет легко добавлять новые команды без изменения существующего кода. Каждая команда отвечает только за свою операцию, что упрощает тестирование и поддержку.

Почему ConsoleListener не статический: Статические методы сложнее тестировать и подменять. Через внедрение зависимости мы можем легко заменить ConsoleListener на мок-объект при тестировании.
